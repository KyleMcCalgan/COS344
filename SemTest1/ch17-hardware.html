Chapter 17

Graphics Hardware & OpenGL

The practical chapter — how the GPU works, how OpenGL exposes it, and how to write actual OpenGL programs with VBOs, VAOs, and GLSL shaders.

Connects to → Ch 9: Pipeline this hardware implements Ch 7: Matrices in vertex shaders Ch 5: Shading in fragment shaders

Section 17.2 — What is Graphics Hardware?

A GPU (Graphics Processing Unit) is a dedicated chip optimised for processing 3D geometry and rendering images. It consists of:

  • Thousands of small computing cores — unlike a CPU's few large cores
  • High parallelism — thousands of threads running simultaneously
  • Dedicated memory (VRAM) — fast access to vertex/texture data

GPUs are also used outside graphics: AI/ML training, scientific simulation, cryptocurrency — because their massively parallel architecture suits any workload that can be decomposed into many identical small operations.

Section 17.3 — Heterogeneous Multiprocessing

A modern system has two distinct processors working together:

CPUGPU
Cores Few (4–32), large and complex Thousands, small and simple
Best at Sequential logic, branching, OS tasks Massively parallel identical operations
Role Main program logic, setup, draw calls Vertex transforms, fragment shading

The CPU sends geometry data and commands to the GPU over a bus (PCIe). The GPU renders to a framebuffer which is then displayed on screen.

Section 17.4 — Buffers, State, and Shaders

The OpenGL programming model is built around three concepts:

  • Buffers — blocks of GPU memory holding vertex data, indices, etc.
  • State — settings that control how OpenGL renders (depth test, blend mode, active shader, etc.)
  • Shaders — small programs that run on the GPU at different stages of the pipeline

Section 17.5 — OpenGL State Machine

OpenGL is a state machine. You set state, then draw. The draw call uses whatever state is currently active.

Enabling capabilities

OpenGL State Functions
// Enable depth testing (z-buffer — see Ch 9)
glEnable(GL_DEPTH_TEST);
glDepthFunc(GL_LESS);  // draw if closer (default)

// Other common depth functions:
// GL_NEVER   GL_LESS    GL_EQUAL   GL_LEQUAL
// GL_GREATER GL_NOTEQUAL GL_GEQUAL GL_ALWAYS

// Disable depth testing
glDisable(GL_DEPTH_TEST);

The render loop

Main Render Loop (GLFW)
do {
    // Clear colour and depth buffers each frame
    glClear(GL_COLOR_BUFFER_BIT | GL_DEPTH_BUFFER_BIT);

    // ... draw your scene here ...

    glfwSwapBuffers(window);  // swap front/back buffers
    glfwPollEvents();         // handle keyboard/mouse

} while (glfwGetKey(window, GLFW_KEY_SPACE) != GLFW_PRESS
      && glfwWindowShouldClose(window) == 0);
💡
Double buffering

OpenGL renders into a "back buffer" while the "front buffer" is displayed. glfwSwapBuffers swaps them, preventing screen tearing.

Section 17.7 — Geometry & Primitives

Vertex attributes (position, colour, normal, etc.) are stored in arrays before being sent to the GPU. The GPU's basic geometric primitives are:

PrimitiveDescription
PointsEach vertex rendered as a point.
LinesUsed for wireframes.
TrianglesUsed for filled polygons. The fundamental primitive.

Triangle primitive modes

ModeVertex groupingExample (a,b,c,d,e,f)
GL_TRIANGLESEvery 3 vertices = one triangleΔ(a,b,c), Δ(d,e,f)
GL_TRIANGLE_STRIPEach new vertex + previous 2 = new triangleΔ(a,b,c), Δ(b,c,d), Δ(c,d,e)
GL_TRIANGLE_FANEach new vertex + first vertex + previous = new triangleΔ(a,b,c), Δ(a,c,d), Δ(a,d,e)
🎯
Triangle winding order

The order of vertices matters! Triangle vertices must be ordered so that the outward-facing normal faces outward (typically counter-clockwise when viewed from outside). This is used for back-face culling.

Line primitive modes

ModeDescription
GL_LINESPairs of vertices: (a,b), (c,d), ...
GL_LINE_STRIPConnected path: a→b→c→d
GL_LINE_LOOPClosed path: a→b→c→d→a

Section 17.8 — Shaders (GLSL)

Modern OpenGL (3.3+) requires shaders. No rendering works without at least:

  • One vertex shader — processes each incoming vertex
  • One fragment shader — processes each rasterized fragment (pixel)

17.8.1 — Vertex Shader

GLSL — Minimal Vertex Shader
#version 330 core
layout(location = 0) in vec3 in_Position;

void main(void) {
    gl_Position = vec4(in_Position, 1.0);
}
GLSL — Vertex Shader with MVP Transform (Ch 7)
#version 330 core
layout(location = 0) in vec3 in_Position;
layout(location = 1) in vec3 in_Normal;

uniform mat4 model;
uniform mat4 view;
uniform mat4 projection;
uniform mat3 normalMatrix;   // = transpose(inverse(model))

out vec3 fragPos;
out vec3 fragNormal;

void main(void) {
    vec4 worldPos = model * vec4(in_Position, 1.0);
    fragPos = vec3(worldPos);
    fragNormal = normalize(normalMatrix * in_Normal);
    gl_Position = projection * view * worldPos;
}

17.8.2 — Fragment Shader

GLSL — Fragment Shader with Blinn-Phong (Ch 5)
#version 330 core
in vec3 fragPos;
in vec3 fragNormal;

uniform vec3 lightPos;
uniform vec3 viewPos;
uniform vec3 lightColor;
uniform vec3 objectColor;

out vec4 FragColor;

void main() {
    vec3 n = normalize(fragNormal);
    vec3 l = normalize(lightPos - fragPos);
    vec3 v = normalize(viewPos - fragPos);
    vec3 h = normalize(l + v);  // half vector (Blinn-Phong)

    // Ambient
    float ka = 0.1;
    vec3 ambient = ka * lightColor;

    // Diffuse (Lambertian)
    float diff = max(dot(n, l), 0.0);
    vec3 diffuse = diff * lightColor;

    // Specular (Blinn-Phong)
    float ks = 0.5;
    float p = 32.0;  // shininess
    float spec = pow(max(dot(n, h), 0.0), p);
    vec3 specular = ks * spec * lightColor;

    vec3 result = (ambient + diffuse + specular) * objectColor;
    FragColor = vec4(result, 1.0);
}

Swizzling

GLSL — Swizzling
vec3 position = vec3(1.0, 2.0, 3.0);
vec2 xy = position.xy;     // (1.0, 2.0)
vec3 zyx = position.zyx;   // (3.0, 2.0, 1.0)
float z = position.z;      // 3.0

// Color swizzling (same indices, different names)
vec4 color = vec4(1.0, 0.5, 0.2, 1.0);
vec3 rgb = color.rgb;  // equivalent to color.xyz

Section 17.9 — Vertex Buffer Objects (VBO)

A VBO is a buffer in GPU memory that holds vertex data (positions, colours, normals, etc.).

C++ — Creating and Filling a VBO
float vertices[] = {
    // x      y      z
    -0.5f, -0.5f, 0.0f,   // vertex 0
     0.5f, -0.5f, 0.0f,   // vertex 1
     0.0f,  0.5f, 0.0f    // vertex 2
};

GLuint VBO;
glGenBuffers(1, &VBO);           // create 1 buffer, get its ID
glBindBuffer(GL_ARRAY_BUFFER, VBO);   // make it the active buffer
glBufferData(GL_ARRAY_BUFFER,         // upload data to GPU
    sizeof(vertices), vertices, GL_STATIC_DRAW);

Section 17.10 — Vertex Array Objects (VAO)

A VAO stores the layout description of vertex data — which attribute is at which location, what type, stride, offset. It remembers the VBO bindings and attribute pointers, so you only need to bind the VAO to use everything.

C++ — VAO Setup
GLuint VAO, VBO;
glGenVertexArrays(1, &VAO);
glGenBuffers(1, &VBO);

glBindVertexArray(VAO);           // start recording

glBindBuffer(GL_ARRAY_BUFFER, VBO);
glBufferData(GL_ARRAY_BUFFER, sizeof(vertices), vertices, GL_STATIC_DRAW);

// Tell OpenGL: attribute 0 = vec3 (position), tightly packed
glVertexAttribPointer(
    0,              // attribute location (matches layout(location=0))
    3,              // size: 3 floats (vec3)
    GL_FLOAT,       // type
    GL_FALSE,       // don't normalise
    3 * sizeof(float),  // stride (bytes between consecutive vertices)
    (void*)0        // offset of this attribute within vertex struct
);
glEnableVertexAttribArray(0);

glBindVertexArray(0);             // stop recording

// --- At render time ---
glBindVertexArray(VAO);
glDrawArrays(GL_TRIANGLES, 0, 3);  // draw 3 vertices as a triangle
VBO / VAO relationship
  CPU Memory:          GPU Memory (VBO):           VAO:
  ┌──────────┐        ┌──────────────────┐        ┌─────────────────────────┐
  │ vertices │──────► │ [x,y,z, x,y,z,…]│◄───────│ attr 0: VBO, float, 3   │
  │ (float[])│        │                  │        │ attr 1: VBO, float, 3   │
  └──────────┘        └──────────────────┘        │ (stores layout + binding)│
                                                  └─────────────────────────┘
  glBufferData uploads     VBO stores raw data     VAO remembers how to read it
🎯
VAO vs VBO — know the difference

VBO = the raw data buffer on the GPU. VAO = the descriptor/recipe for reading that data. You bind the VAO to draw; the VAO tells OpenGL which VBO to read and how the data is laid out.
